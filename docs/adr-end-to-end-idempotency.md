# ADR: End-to-end idempotency-key flow

*Last modified: 2026-05-03*

## Status

Accepted. Builds on `adr-billing-hot-path-vs-async.md` (the layering rule), `adr-http-ledger-protocol.md` (the LedgerClient wire protocol and its `Idempotency-Key` header), `adr-webhook-security.md` (inbound + outbound webhook contract), `adr-admin-action-audit.md`, `adr-audit-log-v0.md`, `adr-wallet-model.md`, `adr-capacity-rate-limits.md` (rate-limit middleware), and `adr-disaster-recovery-retention.md`. Companion ADRs cover signed quote tokens and EVM reorg resistance.

## Context

A single agent request flows through five layers before money moves:

1. **Agent → proxy**: a 402-redeem flow keyed by the quote-token nonce.
2. **Proxy → LedgerClient**: an HTTPS call with an `Idempotency-Key` header.
3. **LedgerClient → BillingRail (async worker)**: the worker submits usage to the rail.
4. **BillingRail → rail (Stripe / x402 facilitator)**: the rail's own native idempotency-key.
5. **Outbound webhook → customer**: a `Sbproxy-Delivery-Id` per attempt.

Each layer can be retried independently. A network blip between layer 2 and layer 3 must not produce a second on-chain settlement at layer 4. A worker crash between layer 3 and layer 4 must resume cleanly. A customer endpoint that 5xxs at layer 5 must retry without re-emitting upstream events. None of this is automatic; each layer's idempotency-key carrier, derivation rule, and storage TTL has to be pinned.

The LedgerClient layer is pinned in `adr-http-ledger-protocol.md`. The webhook layer is pinned in `adr-webhook-security.md`. The rate-limit middleware reading `Idempotency-Key` is pinned in `adr-capacity-rate-limits.md`. This ADR pins the rest: the cross-layer derivation rule, the per-layer TTL, the quote-token nonce as the root of trust, and the rate-limit interaction.

The implementation lands in idempotency middleware in `sbproxy-modules`, rail-specific idempotency derivation in the `BillingRail` adapters, and the worker queue.

## Decision

### Layer-by-layer carriers and TTLs

| Layer | Idempotency-Key carrier | Storage backend | TTL |
|---|---|---|---|
| Agent → proxy (redeem) | quote-token nonce (ULID) | `quote_tokens` table in Postgres (single-use ledger link) | 1 h (matches quote-token `exp`) |
| Proxy → LedgerClient | `Idempotency-Key` HTTP header | Redis (hot) + Postgres `ledger_idempotency` (durable) | 24 h |
| LedgerClient → BillingRail (worker queue) | derived: `worker_idem = sha256(quote_nonce \|\| rail_name)[:32]` | Postgres `worker_idempotency` table | 30 days |
| BillingRail → rail (Stripe / x402) | rail-native: Stripe `Idempotency-Key`, x402 EIP-3009 `nonce` | rail-controlled | rail-controlled (Stripe: 24 h; x402: per-tx nonce, no expiry) |
| Outbound webhook → customer | `Sbproxy-Delivery-Id` (ULID per attempt; `Sbproxy-Event-Id` stable per event) | Postgres `webhook_deliveries` | 30 days (matches deadletter retention) |

The chain reads top-to-bottom: every key derives from the one above it, so a retry at any layer can resolve to the same final state. The quote-token nonce at the top is the **root of trust**; everything else is deterministic from it.

### Cross-layer key derivation

Every non-root key is derived deterministically from the layer above so that a process crash plus restart can reconstruct the chain without storing the keys.

```
quote_nonce          := ULID generated at quote-token issuance
ledger_idem_key      := ULID generated by the proxy on first ledger call;
                        stored in flight-state until layer 2 succeeds.
                        Reused across ledger retries.
worker_idem_key      := sha256(quote_nonce || ":" || rail_name).hex()[:32]
rail_native_key      := per rail (see below)
delivery_id          := ULID generated per webhook attempt
event_id             := ULID stable per logical event
```

Rail-native derivation:

| Rail | Native key | Derivation |
|---|---|---|
| Stripe metered usage | Stripe `Idempotency-Key` header | `worker_idem_key` (passed through unchanged) |
| Stripe refund | Stripe `Idempotency-Key` header | `sha256(redemption_id \|\| ":refund:" \|\| amount).hex()[:32]` |
| x402 EIP-3009 | `transferWithAuthorization.nonce` | First 32 bytes of `worker_idem_key` interpreted as a `bytes32` |
| MPP rail callback | `Sb-Mpp-Idempotency` header | `worker_idem_key` |

The discipline: every derivation is a SHA-256 of well-defined inputs joined by a literal `":"` separator. The 32-character truncation is for rails that limit key length (Stripe caps at 255, but worker-internal keys benefit from being short for log readability). SHA-256's collision resistance is overkill for the input space (ULIDs are already 128-bit unique), but the SHA-256 step removes any theoretical privacy leak from the upstream nonce being visible in rail-side logs.

The rule, stated as one line:

> The quote-token nonce at layer 1 is the root of trust; every downstream idempotency key is `sha256(parent_key || ":" || layer_qualifier)` truncated to the layer's required length. A worker that crashes recomputes the chain from the persisted `quote_nonce`; no idempotency key is ever lost as long as the quote-token row survives.

### The quote-token nonce as root

The `quote_tokens` Postgres table is the single source of truth for "this quote was redeemed exactly once". The schema is:

```sql
CREATE TABLE quote_tokens (
    quote_nonce        TEXT PRIMARY KEY,        -- ULID
    workspace_id       TEXT NOT NULL,
    issued_at          TIMESTAMPTZ NOT NULL,
    expires_at         TIMESTAMPTZ NOT NULL,    -- iat + TTL (default 1h)
    price_micros       BIGINT NOT NULL,
    currency           TEXT NOT NULL,
    shape              TEXT NOT NULL,
    route              TEXT NOT NULL,
    redeemed_at        TIMESTAMPTZ,             -- null until first redeem
    redemption_id      TEXT,                    -- link to redemptions table
    -- ...
);

CREATE UNIQUE INDEX quote_tokens_redemption ON quote_tokens (redemption_id) WHERE redemption_id IS NOT NULL;
```

A redeem call at layer 1 is permitted only if `redeemed_at IS NULL`. The proxy attempts an atomic `UPDATE quote_tokens SET redeemed_at = now(), redemption_id = $1 WHERE quote_nonce = $2 AND redeemed_at IS NULL RETURNING ...`. A second redeem of the same nonce sees zero rows affected and returns `LedgerError::Permanent` with `code = "ledger.token_already_spent"`.

This single-row guarantee is what makes the rest of the chain safe. Even if a buggy agent client sends ten parallel redeems with the same quote nonce, only one will get a `redemption_id` row; the other nine see `token_already_spent` and the chain never propagates a second worker-queue entry.

### Replay window

The longest TTL in the chain is 30 days at the worker layer. The reasoning:

- **Layer 1 (1 h)**: quote-token expiry. Beyond 1 h the price is stale; the agent should re-quote.
- **Layer 2 (24 h)**: ledger idempotency. Long enough to cover any reasonable network partition; short enough to bound storage.
- **Layer 3 (30 days)**: worker queue. This is the load-bearing TTL. A worker that crashes and is offline for two weeks must still resume without duplicating a Stripe submission. 30 days bounds the worst-case operator outage scenario; the data class is "audit-relevant" with 30-day retention, which aligns.
- **Layer 4 (rail-controlled)**: Stripe's window is 24 h, x402's nonce never expires (it is on-chain). The worker layer's 30-day window is a strict superset of Stripe's; if Stripe forgot the key, the worker still has the local "this was already submitted" row and short-circuits on its own.
- **Layer 5 (30 days)**: webhook deadletter. The customer can retry deliveries from the deadletter for up to 30 days; the `Sbproxy-Event-Id` stays stable so customer-side dedup remains correct.

The 30-day worker TTL is the **floor**, not the ceiling. Operators with longer disaster-recovery windows (cold-restore from backup that took 45 days to discover) can extend; the cleanup sweeper reads the value from config. The default is 30 days because that matches the audit retention class and is operationally tractable.

The reason the floor is not "indefinite": indefinite storage of idempotency rows means one Postgres row per request forever. At 1k rps on a busy workspace that is 86 M rows per day, 31 B rows per year. The 30-day cap caps the table at around 2.6 B rows for a busy workspace, which is large but partitionable. Indefinite would not be.

### Cleanup sweeper

A cleanup job runs every 24 h (configurable: `idempotency.sweep_interval_hours`). It deletes rows from each layer's idempotency table where `created_at < now() - retention`. Per layer:

- `ledger_idempotency` (24 h retention): rows older than 24 h. The response cache is no longer needed.
- `worker_idempotency` (30 days retention): rows older than 30 days. The rail-side idempotency window has long expired by then.
- `webhook_deliveries` (30 days): deadletter retention.

The sweeper uses `DELETE ... WHERE created_at < $1 LIMIT 10000` in a tight loop with a 100 ms pause between batches to avoid head-of-line blocking on Postgres. The data class is audit-relevant; the deletion itself does NOT emit an audit envelope (the rows are not state-bearing in the audit sense; they are caches of completed computations). The aggregate counts emit a metric (`sbproxy_idempotency_swept_total{layer}`) so operators can see the sweeper is making progress.

### Rate-limit interaction

The rate-limit middleware reads `Idempotency-Key` and bypasses the bucket on a hit. Concretely, the middleware sequence on every inbound paid request is:

1. Parse `Idempotency-Key` header (if present).
2. If a key is present AND the cached response is non-expired AND the request body hash matches the cached entry's body hash: return the cached response immediately. **Do not consume a token bucket slot.** Emit `sbproxy_idempotency_hits_total{layer, workspace_id}`.
3. If the body hash does NOT match the cached entry: return 409 `ledger.idempotency_conflict`. **Consume one slot** (this is a real work attempt that fails fast; it is rate-limit-relevant because a flood of conflicting keys is a known attack pattern).
4. If the key is absent or the cache misses: consume a slot per the normal rate-limit flow.

The reason an idempotent retry must NOT consume a slot: by definition no new work is being done. The agent is retrying a request the proxy already serviced; the proxy's local cache returns the response without touching the wallet, the ledger, or any rail. Counting it against the bucket would punish the agent for the proxy's own retry behaviour.

The reason a conflict (case 3) DOES consume a slot: a flood of conflicting `Idempotency-Key` retries is a credible DoS pattern (an attacker tries to fill the cache with junk keys). The rate-limit must apply.

This semantics matches Stripe's idempotency-key behaviour (cached responses do not consume API rate-limit budget). It is documented in `docs/idempotency.md` for customer-facing clarity.

### Storage layout

```sql
-- Layer 2: proxy → LedgerClient cache.
-- Read-heavy on retry; bounded by 24h sweep.
CREATE TABLE ledger_idempotency (
    idempotency_key   TEXT PRIMARY KEY,
    workspace_id      TEXT NOT NULL,
    request_body_hash TEXT NOT NULL,           -- sha256:<base64>
    response_status   INT NOT NULL,
    response_body     BYTEA NOT NULL,          -- gzipped JSON envelope
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ledger_idempotency_created ON ledger_idempotency (created_at);

-- Layer 3: worker queue dedup.
-- Read-once on worker resume; bounded by 30-day sweep.
CREATE TABLE worker_idempotency (
    worker_idem_key   TEXT PRIMARY KEY,        -- sha256(quote_nonce||rail)
    quote_nonce       TEXT NOT NULL,
    rail_name         TEXT NOT NULL,
    rail_response     JSONB,                   -- null until rail returns
    rail_status       TEXT NOT NULL,           -- pending | submitted | confirmed | failed
    submitted_at      TIMESTAMPTZ,
    confirmed_at      TIMESTAMPTZ,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (quote_nonce, rail_name)
);
CREATE INDEX worker_idempotency_status ON worker_idempotency (rail_status, created_at);
CREATE INDEX worker_idempotency_created ON worker_idempotency (created_at);
```

The `worker_idempotency` table doubles as the worker queue's "checkpoint" log: a worker resuming after a crash reads `WHERE rail_status = 'pending'` to find work. The rail-native key (Stripe's idempotency-key, x402's nonce) is reproducible from `worker_idem_key` plus the rail-specific derivation, so the worker does not need to persist the rail-native key separately.

Layer 4 (rail-side) and layer 5 (webhook deliveries) have their storage pinned in `adr-webhook-security.md`.

### Hot-path Redis cache for layer 2

The Postgres-backed `ledger_idempotency` table is durable but a network round-trip for every retry. At a 1k rps default, a 5% retry rate at 50 us Postgres lookup is 2.5 ms aggregate latency, which is fine. But for high-rps workspaces (the 10k hard ceiling) a Redis cache in front of the Postgres table cuts the read to around 200 us typical.

The cache is read-through: lookup hits Redis first, falls back to Postgres on miss, populates Redis on Postgres hit. Writes go to Postgres first (durable), then Redis (cache). A Redis outage degrades to Postgres-only with measurable but acceptable latency increase; a Postgres outage is an `SLO-LEDGER` incident regardless.

This caching strategy mirrors the webhook replay cache (Redis primary + Postgres fallback) and is the same shape used in the wallet hot store. Operators with no Redis run Postgres-only.

### Trace-span attributes

Every layer's idempotency activity surfaces as a span attribute on the request trace:

```
sbproxy.idempotency.quote_nonce        = "01J..."
sbproxy.idempotency.ledger_key         = "01J..."
sbproxy.idempotency.worker_key         = "abc123..." (truncated 8-char hex)
sbproxy.idempotency.layer1_hit         = false   (quote not previously redeemed)
sbproxy.idempotency.layer2_hit         = true    (ledger response cached)
sbproxy.idempotency.layer3_hit         = false   (worker submission new)
sbproxy.idempotency.delivery_id        = "01J..."
```

The middleware sets these on the request span. The trace lets a debugger reconstruct exactly which layer cached and which actually called downstream, which is the load-bearing observability for "is the chain working as designed?" investigations.

### What this ADR does NOT decide

- The exact body-hash algorithm (this ADR uses SHA-256). Future signature-prefix conventions may upgrade.
- Per-customer custom retry behaviour at layer 5; that is per-subscription config.
- Cross-region replication of idempotency tables (active-active idempotency requires a CRDT or distributed lock that this ADR does not commit to). Single-region by default.
- Rail-native idempotency-key formats for new rails (Lightning, custom MPP rails). Each rail's adapter ADR pins its own derivation under the same SHA-256 chain rule.

## Consequences

- A retry at any layer resolves to the same outcome. Network blip between proxy and ledger: layer 2 hit, layer 3 untouched, no double-charge. Worker crash mid-submission: layer 3 short-circuits on resume because the row exists; layer 4's rail-native key matches Stripe's cache, second submission is a no-op. Customer endpoint 5xx: layer 5 retries with same `Sbproxy-Event-Id`, customer dedupes.
- The 30-day worker TTL is the de facto "how long can the operator be offline before idempotency degrades?" budget. Within that window, recovery is automatic.
- The deterministic SHA-256 chain means a worker can resume from `quote_nonce` alone. The persistence requirements collapse to "the `quote_tokens` row survives"; that row is in the same Postgres database as everything else, so the durability story is one database's durability.
- Rate-limit semantics match customer expectations: idempotent retries do not punish the agent. Conflict floods do.
- The Redis cache for layer 2 keeps the hot path under 1 ms p99 for retry hits at 10k rps. Operators without Redis pay a few ms but the chain is still correct.
- The cleanup sweeper's 24 h cadence keeps the tables bounded. Operators see `sbproxy_idempotency_swept_total` ticking up; operational health dashboards show table growth flat.
- The cross-layer derivation makes the chain auditable: an investigator with a `quote_nonce` can compute every downstream key and find the corresponding row in every layer's table. No layer's key is "free-floating" without a derivation back to the root.

## Alternatives considered

**Use the LedgerClient `Idempotency-Key` as the worker-layer key directly.** Rejected. The ledger key is per-attempt ("a retry of a network failure for the same logical operation reuses the key"), but the proxy generates it freshly on each first attempt. Across two genuinely independent agent requests with the same quote-token (which is correctly rejected at layer 1), the ledger keys differ. Using the ledger key at the worker layer would mean the worker cannot dedup the layer-1-rejected case from the layer-1-accepted case, which is the wrong shape.

**Generate a UUID at the worker layer and persist it without derivation.** Rejected. A worker that crashes between "wrote queue row with UUID" and "called rail with UUID" loses the UUID on restart and would generate a new one on resume, double-submitting. Deterministic derivation from `quote_nonce` (which is durable in `quote_tokens`) avoids this.

**Skip layer 3 entirely and let the rail's native idempotency be the only dedup.** Rejected. Stripe's 24 h window is too short to cover all worker-outage scenarios; x402's nonce is durable but the worker still needs a local "this was already submitted" record so it doesn't re-broadcast a tx that was already on-chain. The worker layer's row is the operator-side proof of "we did the work"; the rail's native dedup is the rail-side proof. Both are needed.

**Single global TTL across all layers.** Rejected. The TTLs are doing different jobs: layer 1 caps quote staleness, layer 2 caps the proxy's retry window, layer 3 caps the worker's outage tolerance, layer 5 caps customer-endpoint deadletter age. Forcing them to be the same number either over-stores at layer 1 or under-stores at layer 3.

**Charge an idempotent retry against the rate-limit bucket.** Rejected. By definition no new work is being done; charging the bucket would punish the agent for the proxy's retry semantics. Stripe's behaviour matches this; aligning is the right call.

**Indefinite storage of idempotency rows.** Rejected. Storage cost grows linearly with request volume forever; partitionability does not solve the long-tail cost. The 30-day worker floor covers all realistic operator-outage scenarios.

**Use HMAC instead of SHA-256 for the cross-layer derivation.** Considered. HMAC is keyed; SHA-256 is unkeyed. Since the inputs (`quote_nonce` + layer qualifier) are not secrets and the goal is deterministic derivation, SHA-256 is sufficient. HMAC would require a key-management story for what is not a security primitive; rejected as needless complexity.
