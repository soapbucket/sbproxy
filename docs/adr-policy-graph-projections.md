# ADR: Policy-graph projections as well-known documents

*Last modified: 2026-05-03*

## Status

Accepted. Builds on `adr-multi-rail-402-challenge.md`, `adr-quote-token-jws.md`, `adr-schema-versioning.md`, `adr-admin-action-audit.md`, and the content negotiation ADR `adr-content-negotiation-and-pricing.md`.

## Context

The operator's `ai_crawl_control` policy is the single source of truth for who can crawl what, at what price, with what content-shape constraints, and under what license terms. That information currently exists only in the proxy's runtime compiled config; agents have no way to discover it before hitting a 402 wall.

The AI standards landscape has converged on four well-known discovery documents:

- `robots.txt`, extended for AI per IETF draft-koster-rep-ai (agent-class disallow, crawl-delay, pay-per-crawl hints).
- `llms.txt` and `llms-full.txt`, a convention proposed by Anthropic, Mistral, and others for LLM-readable site descriptions and content listings.
- `/licenses.xml`, RSL 1.0 (RSL Collective, December 2025), a machine-readable statement of which content is licensed for what AI use.
- `/.well-known/tdmrep.json`, W3C TDMRep, stating text-and-data-mining rights per EU DSM Directive Article 4.

This ADR ships all four as live documents generated from the compiled policy graph. They are not static files; they are projections of the operator's config, regenerated on every config reload, and served from the request pipeline.

The key design question is how to keep the four documents consistent with each other and with the live policy without introducing a separate sync process, a separate config store, or a separate pipeline.

## Decision

### Input: the compiled policy graph

The policy graph for each projection is the set of `(origin_hostname, tier_list, content_signal, rails)` tuples extracted from every `CompiledOrigin` whose `policy_configs` list contains at least one `ai_crawl_control` entry. The compiler already builds `CompiledOrigin` from YAML at startup and on every hot reload; projections consume the same data with no additional parsing.

Concretely, the input to the projection engine is:

```rust
pub struct ProjectionInput<'a> {
    /// Reference to the compiled config. The projection engine walks
    /// `config.origins`, filters for origins with ai_crawl_control policies,
    /// and extracts pricing tiers, content_signal values, and rail sets.
    pub config: &'a CompiledConfig,
    /// Version hash of the config. Used as the projection cache key.
    /// Derived from the same hash the hot-reload path computes when
    /// deciding whether the new config differs from the in-flight one.
    pub config_version: u64,
}
```

No additional data structures are introduced in the compiled config. The `Tier` array and `ContentShape` enum are already in `ai_crawl.rs`; the `Content-Signal` value is extracted from the origin's `extensions` map (the `content_signal` key).

All four documents are pure OSS; an extension override is a stub plugin hook described below.

### Projection documents

**`robots.txt`.** Served at `/robots.txt` on every hostname that has an `ai_crawl_control` policy. Format follows IETF draft-koster-rep-ai (AI-extended robots.txt):

```
# Generated by SBproxy. Do not edit.
# Config version: <hash>

User-agent: GPTBot
Disallow: /premium/*
Crawl-delay: 1
# SBproxy-AI-Extension: pay-per-crawl price=0.001 currency=USD shape=html

User-agent: *
Disallow:
```

One `User-agent:` stanza per agent class that has a Disallow rule (derived from tiers with `price.amount_micros > 0` against that agent). The `SBproxy-AI-Extension` comment lines carry pricing metadata for cooperative crawlers. The extension comment format is not yet an IETF standard; the prefix `SBproxy-AI-Extension:` is used to make it clearly non-standard, pending standardisation.

**`llms.txt` and `llms-full.txt`.** Served at `/llms.txt` and `/llms-full.txt`. Format follows the Anthropic / Mistral convention: YAML-like metadata block followed by a Markdown site description. `llms.txt` is a concise summary; `llms-full.txt` includes the full tier listing.

The implementation emits:
- `# sitename:` from the origin hostname.
- `# version:` from the config version hash.
- `# payment:` the top-level price in the catch-all tier (or "pay-per-request" for complex tier sets).
- `# shapes:` comma-separated list of offered `ContentShape` values.
- A Markdown body listing the priced routes (for `llms-full.txt`).

**`/licenses.xml`.** Served at `/licenses.xml`. RSL 1.0 format. The RSL document asserts the license terms for AI ingestion of this origin's content. Each `<license>` element in the document corresponds to one origin-level `Content-Signal` value:

| `Content-Signal` value | RSL `<ai-use>` assertion |
|---|---|
| `ai-train` | `<ai-use type="training" licensed="true" />` |
| `ai-input` | `<ai-use type="inference" licensed="true" />` |
| `search` | `<ai-use type="search-index" licensed="true" />` |
| absent | `<ai-use type="training" licensed="false" />` (default-deny) |

The RSL URN for the document is `urn:rsl:1.0:<origin_hostname>:<config_version_hash>`. The URN is the value the JSON envelope uses in the `license` field.

**`/.well-known/tdmrep.json`.** Served at `/.well-known/tdmrep.json`. W3C TDMRep JSON format. One entry per priced route:

```json
{
  "version": "1.0",
  "generated": "<RFC3339 timestamp>",
  "policies": [
    {
      "location": "/articles/*",
      "mine-type": ["text/html", "text/markdown"],
      "right": "research",
      "license": "https://sbproxy.dev/licenses/<origin_hostname>"
    }
  ]
}
```

The `right` field maps `Content-Signal` to TDMRep: `ai-train` produces `"right": "train"`, `ai-input` and `search` produce `"right": "research"`, absent signal produces no entry (no right asserted = right reserved).

### In-memory projection cache

The four documents are computed once per config version and cached in a single `Arc<ProjectionDocs>`:

```rust
pub struct ProjectionDocs {
    /// Config version hash this snapshot was computed from.
    pub config_version: u64,
    /// Per-hostname robots.txt bodies.
    pub robots_txt: HashMap<CompactString, Bytes>,
    /// Per-hostname llms.txt bodies.
    pub llms_txt: HashMap<CompactString, Bytes>,
    /// Per-hostname llms-full.txt bodies.
    pub llms_full_txt: HashMap<CompactString, Bytes>,
    /// Per-hostname /licenses.xml bodies.
    pub licenses_xml: HashMap<CompactString, Bytes>,
    /// Per-hostname /.well-known/tdmrep.json bodies.
    pub tdmrep_json: HashMap<CompactString, Bytes>,
}
```

The cache is stored in the same `ArcSwap` pattern used by the pipeline: a `OnceLock<ArcSwap<ProjectionDocs>>` in `sbproxy-core::reload`, swapped atomically on every config reload. The reload hook calls the projection engine after `load_pipeline` completes and before the old pipeline guard is released. This ensures the projection cache and the live policy are always at the same config version.

Contention model: readers load the `Arc<ProjectionDocs>` from the `ArcSwap` with no lock. Writers (the reload path) call `ArcSwap::store` once per reload. There is no writer-reader conflict because `ArcSwap` is lock-free for readers. The cost per projection-serving request is one atomic load and one clone of the `Arc` (the body bytes are shared).

### Refresh triggers

Projections regenerate exactly once per config reload, not on first request and not on a timer. This matches the existing `load_pipeline` / `current_pipeline` pattern in `crates/sbproxy-core/src/reload.rs`. The reload path already computes a config version hash; the projection engine receives that hash and uses it as the cache key to detect stale hits (the hot path checks `cache.config_version == current_version` before serving; on mismatch it triggers a synchronous regeneration, which should be impossible in steady state but is the safe fallback).

### Extension plugin hook

The four projections are pure OSS. An extension override allows downstream code to emit additional per-agent-class entries in `robots.txt` derived from a reputation feed:

```rust
/// Optional override for the projection engine. The OSS engine
/// calls this after computing the base projection. Extension code may
/// extend or replace individual fields. Returns None to indicate no override.
pub trait ProjectionOverride: Send + Sync {
    fn extend_robots_txt(&self, hostname: &str, base: &mut String);
    fn extend_llms_txt(&self, hostname: &str, base: &mut String);
    // ...
}
```

The OSS binary constructs a `NoOpProjectionOverride` that returns immediately. Downstream code registers a concrete implementation via the existing `inventory`-based registry pattern. The trait lives in `sbproxy-plugin`.

The OSS distribution ships the `NoOpProjectionOverride` only. The override trait is declared but not implemented. The four projection engines do not need any external data.

### Operator preview via CLI

Operators preview a projection before pushing config with the `sbproxy projections render` CLI subcommand:

```
sbproxy projections render --kind robots --config ./sb.yml
sbproxy projections render --kind llms --config ./sb.yml
sbproxy projections render --kind licenses --config ./sb.yml
sbproxy projections render --kind tdmrep --config ./sb.yml
```

The command loads the YAML config, compiles it (same path as the proxy startup), runs the projection engine on the compiled output, and writes the document to stdout. No admin endpoint is added at this stage. The CLI subcommand lives in the `sbproxy` binary crate. An admin endpoint is deferred until the admin API surface is broader.

The CLI uses the same `ProjectionEngine` struct the proxy runtime uses, so preview output is byte-for-byte identical to the document the proxy would serve for that config.

### Audit trail

Each projection regeneration emits an `AdminAuditEvent` per `adr-admin-action-audit.md` with:
- `action: AuditAction::PolicyProjectionRefresh`
- `target: AuditTarget { kind: "PolicyProjection", id: "<hostname>/<projection_kind>/<config_version>" }`
- `after: Some(json!({ "doc_hash": "<sha256_of_body>", "byte_len": <len> }))`
- `schema_version: 0` (current v0)

The audit event is emitted once per (hostname, projection kind) pair per reload. For an operator with 10 origins, one reload emits 40 audit events (10 origins x 4 projection kinds). The audit sink handles this volume without backpressure.

The event carries `target_kind = "PolicyProjection"` and the document hash to allow external auditors to verify that the served document matches what was recorded at reload time.

## Consequences

- Projections are always consistent with the live policy: a reload that changes a tier price regenerates all four documents atomically in the same reload path. There is no window where the policy and the documents diverge.
- The `ArcSwap` pattern for `ProjectionDocs` mirrors the pipeline store pattern exactly. Readers pay one atomic load; writers pay one store per reload. No additional locks.
- The CLI preview command gives operators a safe way to audit their site's public discovery posture before pushing config. This is a user-facing safety net.
- All four projections are OSS. External feed data is a future extension; the plugin hook is declared now so a future change does not need to refactor the projection engine's call sites.
- The audit trail adds 40 events per reload for a 10-origin deployment. This is acceptable; audit storage is append-only and the events are small JSON lines.

## Alternatives considered

**Generate projections on first request (lazy).** Rejected. First-request generation means the first agent to arrive triggers a potentially expensive computation and then races with subsequent readers. Eager generation at reload time pushes the cost to a non-hot-path event.

**Store projections in a separate KV store (Redis / Pebble).** Rejected. The `ArcSwap<ProjectionDocs>` provides sub-microsecond reads from in-process memory. Adding a network or disk hop for a document that changes only on config reload is unnecessary complexity.

**Serve projections from a separate admin endpoint (not at the canonical well-known URL).** Rejected. The well-known URLs (`/robots.txt`, `/.well-known/tdmrep.json`) are the agent's discovery entry point. An admin-only endpoint defeats the purpose.

**Admin HTTP endpoint for preview instead of CLI.** Considered. Rejected because the admin server is not yet stable enough to add a new preview endpoint. CLI is safer and has the advantage of being usable in CI pipelines without a running proxy.

## Open questions

1. **Per-agent `robots.txt` stanzas from an external feed.** The override trait is declared but not implemented. When external-feed-driven disallow stanzas land, does the `extend_robots_txt` hook receive a structured `Vec<AgentClassEntry>` or a raw string? Recommend structured.
2. **RSL URN namespace.** The `urn:rsl:1.0:<origin_hostname>:<config_version_hash>` is proposed here but not validated against the RSL Collective's published namespace registry. Before the projection ships, the RSL 1.0 spec should be checked for required URN structure. If the spec mandates a different pattern, the projection should follow the spec.
